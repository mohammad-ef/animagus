  ## ** Getting Started**

 First set up your environment.  Install Node (minimum v16) to run build scripts. Python (3.9+) is helpful for lint checks but not strictly required for the build. A code or editor you'll like.

 To get a local clone, simply use the command: `git clone https://github.com/user/Awesome. git`

 After a successful clone, use the command `npm install` (or `yarn install`, depending) in the directory to ensure all project's dependencies will install as needed for building or compiling.

 We use `main` as primary source branch for the codebase. It is important when making a new change to not submit it right to that but rather to a dedicated, unique feature branches instead.

 For local build, type `npx webpack` and you will have a build artifact that reflects what can eventually deployed to your server.

 ## 5. Reporting Bugs

When finding a potential problem or flaw please report to Github Issue. Please be detailed and concise, explaining what steps will lead us there and how often we should encounter them

  A good bug report includes, clear explanation and reproduction of error steps as they are performed, system specifications (Operating System), browser/application/platform used for the test

 Please be as precise and succinct with reporting issues as the quicker it gets understood it's the faster you see an eventual remedy. Include screenshots where available and provide clear instructions.

 Always include any applicable log output or traceback data which is available in the logs. Adding the log file provides valuable insights that help to isolate potential problem points faster!

## 7. Pull Request Process

 To make the best and fastest contributions please fork project. Once your feature development complete use command: `git checkout -b feature/my-feature`
	
 Create branch with name like feature/[myfeature]. This allows us separate feature-driven progress while preventing any interfernce on other developers. Make commits small to keep track

 After completing feature and test your change create Pull Request against main (usually) or another branch you are targeting to incorporate change for evaluation by project contributors

 Pull Request review will take some time to allow time. Address reviewer concerns before merge approval for your PR, it will speed your contributions up.

## 9. Testing Guidelines

 Run test before submit any PR by issuing commands, `npm test` in shell/term. This ensures code quality as new feature is tested as per expectations

## 11. Commit Message Guidelines

 Commit Messages need adhere strict conventions to allow efficient tracking, code reviews & history comprehension, as this will ensure our repository builds and functions well

 Commit Messages will conform using "conventional commits." Prefix each with the action like  `feat:`, `fix:`, `docs:` followed succinct explanation of change 

 Clear and precise commits allow developers track the changes easily over history; these contribute greatly toward understanding and resolving problems.

 Proper messages make diff tracking and automatic version bumps easier as per the commit types (major/minor). These messages must follow proper format or your contributions could get rejected for non-compliance!

## 12. Branching Model

The current approach to versions relies primarily with Trunk-Based Branching, in short it minimizes branching for a simple workflow

 Developers work mostly against primary branch which simplifies merge cycles. Feature Branch is made as need for isolated work or significant refactor

 Always branch from 'main'. When creating the features, name branch as a clear and specific descriptor of your work being carried (example :feature/[your feature].).

 Trunk branching reduces conflicts between versions as developers work against latest codebase which promotes continuous code delivery and easier merging practices!
