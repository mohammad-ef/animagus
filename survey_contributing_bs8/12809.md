Whenever encountering a bug while using the project, a concise and detailed report is invaluable.  The more information that you provide, the easier it will be for the team to reproduce and fix the issue effectively! ## 7. The Pull Request Process:  Merging Your Code ü§ù

To submit your changes, fork the repository as mentioned above, create a new branch with a descriptive name (e.g., `feature/add-dark mode` or `fix/issue-123 `), make your changes locally, and commit them, using descriptive commit messages! Then send your pull request! The name of this branch should clearly communicate your intentions!

Commit messages should be concise, explaining the specific change made. Follow our established convention. Ensure that your commit messages are clear, consistent and well documented! You should never submit any commit that isn't clear or properly documented. 

Upon submission, maintainers will review your pull request for style, correctness, and adherence to the project's conventions and standards. You may receive requests for further clarification or additional changes, so be responsive and open to feedback. After approval and successful integration tests, your code will be merged!

Once your changes have been approved and merged, you can safely delete the branch from your local machine. This ensures a clean and organized workflow! The branch has served its purpose!

## 1 1. Commit Message Guidelines:  Clear and Concise üìù

Consistent commit messages are key to understanding the project's history and making it easier to revert or debug changes. Start by including a brief summary followed by a more descriptive explanation as necessary!

Utilize conventional commit guidelines: prefix each commit message with a type (feat, fix, docs, style, refactor, test, chore) to indicate the nature of the change made!
