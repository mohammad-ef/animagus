## **3. Getting Started: Set Up Your Local Environment**

First, fork the project repository! This creates a your own isolated copy that you' can make modifications, submit pull requests to and generally work from independently. Once the repository is forked, clone it locally to your computer using: `git clone <your fork url>`. It's a good practice to keep a local copy and a for of the project for your personal use and contribution workflows respectively. Make your modifications, and then submit pull requests to our central repository!   

Next, make sure you have the necessary tooling. You will ideally require Go (1.20 or later), a text editor and a Git client for tracking your work. You will also need a build tool for your environment of choice to be installed, like make, or go build, to compile the code for your specific operating system to be executable on a machine. Make sure your dependencies are correctly set up before you begin to modify the platform, to reduce any confusion later down the workflow line.

We encourage the utilization of a virtual environment manager to ensure your projects do not impact any existing installations, like `py env`, `conda`. This way your environment will remain as contained as necessary. To install the basic build dependencies for Chrona run `go mod download ` within the directory. This step will grab all of the dependencies for your local development, so the compilation will work out the box, ready for your changes!

The project utilizes a `feature branch` workflow, which is a standard for many open- source project teams. This will ensure we can track all contributions properly. Create new branches for your work by using `git checkout -b feature/<your-work>`, which will create a separate, local branch you can work on in relative isolation, and which you can easily merge into `main` after your work has been validated. 

To start running the local development, you can simply run the `make start`, which will execute the core processes of the software, so you can start to make your changes.  

Finally, familiarize yourself with existing tests, which are available via running ` make test`, which will allow you to get familiar with our test- driven development methodology before you start to change the codebase.

## **5. Reporting Bugs: Help us Identify and Fix** 

When you encounter a bug, please submit a clear and concise report on GitHub Issues. This is critical in making the debugging and fixing process as easy as possible.  

Be sure to include detailed steps to reproduce the bug, including the specific environment, versions installed, and relevant configuration details. Providing a small, self- contained, complete, and runnable (SSCR - a term often heard in the software industry), code example that triggers the bug is immensely helpful, if applicable!  

Also, include the full error logs that you received. Often there is valuable diagnostic information hidden in the logs, that will help us quickly track down the underlying problems - and this allows us to solve the problem as quick as possible. Please include a description of the expected behaviour, what the behaviour was, in addition to the logs and a reproducible example, if possible to provide a comprehensive view of the problem that's occurring for debugging purposes.

We believe the best improvements come from a collaborative process.  To suggest a new enhancement, start a conversation on GitHub Discussions outlining the problem you're trying to solve or a new feature. Describe your proposed solution, and highlight the potential use cases and benefits.

## **7. Pull Request Process: Your Path to Integration**    

To submit a contribution, first, create a branch from your fork, using `git checkout -b feature/<your-feature-name>`. This isolates all of your contributions in a separate branch, which we can then easily merge, or reject if we feel the code isn 't up to the quality of the platform.

Commit your changes with clear and concise messages, that follow conventional commit guidelines (see our Commit Message Guideline). Push your branch to your own fork and then submit a pull request to the main repo via the GitHub UI. This allows us to start reviewing the code, before we integrate your changes.

When submitting, please provide a detailed description of what's in the Pull Request, so we can easily track all the relevant context of your submission. This will make it easier to validate your work, and to merge the PR to the platform.  Ensure all tests pass ( `make test`) before submitting a PR, so you don' t create a new breaking issue, which would have to be fixed as a follow up, adding complexity and delays to the platform's development lifecycle.

Our maintain team will review the changes carefully, and provide feedback or ask questions. We welcome constructive dialogue during this process, which allows us to ensure a high quality platform, by catching all of the potential issues that could be occurring with the new changes. Once the code is approved and merged, it will be part of the next Chrona release.

## **8. Coding Standards: Writing Clean, Readable Code**

We adhere to Go coding conventions (https:// golang.org/doc/effective_go). This includes indentation, naming conventions, comment formatting, and overall code style for a cleaner codebase.  

Linting is essential to enforce these standards, so we use ` golangci-lint` (configured in `.golangci.yml`) to check code against a range of common issues, like formatting, error checking and unused code. Before you submit a pull request, run ` make lint`, this will execute `golangci -lint` on the current codebase, so we catch all of the linting issues before it gets reviewed manually. This ensures a smoother and more pleasant code- review process.

Type checks are also important and we make sure to run ` go vet`, to catch any common type mismatches, that would otherwise lead to unexpected bugs during production. 

## **9. Testing Guidelines: Ensuring Quality and Reliability**

We believe in comprehensive testing, so we encourage the writing of unit, integration and regression tests. This allows us to catch issues earlier in the workflow. 

To run existing tests, you can simply execute ` make test`, within the project directory, which validates the platform' s core functionalities. We encourage the use of a testing framework, such as ` testify` or `ginkgo`, which help make the tests readable and maintainable, and easy to validate. 

When adding new features, be sure to write unit tests to verify the correctness of your code, and ensure the platform is functioning as it's meant to be, after the new functionality has been added to the codebase. This ensures a smoother and more pleasant code- review process

## **11. Commit Message Guidelines: Clear and Informative Logs**

Well-written commit messages are essential for maintaining a clear and understandable project history. Adhere to the convention commits specification (https:// www.conventionalcommits. org/ en/ v1.0.0/) for all your commits.  

Use the following prefixes to indicate the type of change: `feat:` for new features, `fix:` for bug fixes, `docs:` for documentation, `style:` for code formatting changes, `refactor:` for code refactoring (no functional changes), `test:` for adding or modifying tests. This allows us to track what changes were made, what features were included in your contributions and how much time you have invested to make Chrona more functional! 

Concise commit messages help with understanding and tracing code evolution across the timeline.

## **12. Branching Model: Strategic Branch Management**

We follow a trunk based branching strategy, and we encourage the creation of many smaller commits instead of one massive PR! We encourage you to keep a short lifespan and to rebase regularly so there is less conflict between merges in futures pull requests!

New development should originate from `main` to maintain code clarity, as the trunk-based branch will regularly have updates, ensuring you are validating changes that work on top of what others are building - and vice versa!  We encourage all contributions to be validated before they become main stream so the community is not burdened with broken functionality or regressions due to unstable builds, and can continue their development in confidence and comfort.
