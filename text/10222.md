# How We Contribute

Welcome! This documentation outlines how individuals can actively contribute to our project. Your participation is highly appreciated and essential for continuous growth & evolution. Whether you' re a software developer fixing bugs, a technical writer refining our documentation, or someone offering feedback, your contributions make a difference. Every suggestion, fix & addition helps shape it into a better software! 

We believe that a vibrant & welcoming contributor pool is the backbone of a great project. By contributing your expertise, ideas, and hard work, you directly contribute to its success and make it a more valuable resource. This fosters a stronger community and encourages further growth and collaboration. 

It is our strong intent to foster inclusive growth and a collaborative environment for developers from all walks of life. Every contribution is considered a valuable part and we are thankful for all contributors! 


## Our Code of Conduct

Our community values collaboration and respect.  We strive to maintain a supportive and inclusive environment for everyone and encourage positive, collaborative behavior. We welcome feedback and open discussion but require it be given with courtesy and respect for differing experiences, opinions & skillsets.

Please note that any behavior exhibiting prejudice, harassment, intimidation, or any form of discrimination will not be acceptable. Maintainers will investigate violations with discretion to resolve any concerns while ensuring fairness and upholding these community standards to preserve the integrity of the community. 

Violation of our Code of Conduct may be addressed with warnings or, in severe situations, bans from the codebase. Please review this code thoroughly to understand our principles and maintain an environment that is respectful for all contributors.


 ## 1. Getting Started
 To begin your journey with our project, you will start with the setup of your local development environment. This is the first critical step toward contributing meaningfully. Ensure you have necessary prerequisites, such as a recent Git version installed, and a compatible Python distribution (if the project is based on Python).

 The next step is to create a new local branch for your contribution. Do not work directly on `main`/`master`; always branch from it to isolate your changes. This prevents accidental changes to the main codebase, which is important for team-wide consistency.  ` git fetch origin` then  `git checkout -b feature/your-branch-name`. Use your feature or fix naming conventions! 

 To install dependencies, typically ` pip install -r requirements.txt` is necessary. For other project types, follow the instructions listed within the README.md for specific build commands or any pre-installation steps needed before running the local project or tests. Ensure these steps are complete prior to beginning any development work or submitting your pull request.

  After setting up your environment, clone the repository: `git clone <repository_url>`.  `cd` into directory and then run your project setup script (if one exists, check the README for details).  The goal is to get a fully local environment where you can run tests and verify your local setup is running as expected! This confirms a stable and working environment for contributing and resolving problems early.

 We encourage contributors to regularly pull the latest changes from upstream. Use `git pull origin main ` to incorporate the latest updates from the remote repository. This will keep your local branch synced and reduce potential conflicts when integrating your changes into the project.

## 3. Reporting Bugs

Whenever you stumble upon a bug while using the software, it's valuable to create a well-structured bug report that clearly articulates what, where, and why! The more information included, the faster and easier it will be to reproduce and fix. It saves a significant amount of back and forth with developers!   

First, check if someone else has already reported the same issue in the project repository. Avoid duplicates to consolidate troubleshooting efforts! To submit, head to GitHub's 'Issues' section in the repo. 

The report should describe what steps you followed when encountering the bug (clear, detailed, easy to follow reproduction path) along with its effect: unexpected results vs expected, and screenshots. Be specific; vague or unclear reports are more difficult to debug and often go unresolved! Also add relevant versions (operating system, dependencies.. etc.)

If you could provide a simplified example which reliably triggers the issue. Because we can quickly identify & verify the problem, and the resolution will result in fewer issues in other scenarios and platforms! The more reproducible, the easier & faster a team is able to resolve an issue. 


## 4. Suggesting Enhancements

Got a brilliant idea to improve our project? That's awesome! Before implementing, propose the suggestion, giving details around how to use the proposed enhancement and its expected behavior in detail to ensure that you understand our requirements as much as we can clarify yours. It saves development cycles. 

When submitting suggestions on GitHub (or the specified channels, e.g., mailing list, discussion board), clearly outline the *motivation* behind your suggestion (the *why*), what benefits will arise for both existing users and future potential ones (what problem do you intend to solve). Include detailed potential use cases for your enhancement, as much as you understand its implementation and challenges (what are your scenarios, your goals with this addition) .

Our maintainers will review enhancement suggestions with a team and discuss their viability given current goals, resource capacity, design constraints and alignment with roadmap (if one is existing!). This review is essential in prioritization of features! We aim to respond promptly to your request, whether acceptance is in scope now, later or unfortunately outside future scope given project direction!


## 5. Pull Request Process

The pull request process forms the foundation for all external changes, so this documentation explains what the standard workflow consists of. It ensures changes undergo a quality and thorough evaluation before being merged, and it ensures changes align and do not conflict or negatively impact functionality of code in existing code and functionality!

Fork the repository to your personal GitHub account (e.g., `<username>/project-name`), or work on your existing local branch (feature branch from our project!). Always make sure that your branch reflects latest main changes using: `git pull origin main` or appropriate upstream for a clean & up to date environment for working!   Then checkout the feature branch from `git checkout -b feature/your-branch-name`. Use a consistent prefix like: "feature-" (for adding new capabilities), or a "fixing" or equivalent, prefix that clearly describes change. 

Submit a pull request through GitHub (from your fork), choosing your relevant target base (main / develop). Provide a succinct & concise description that describes exactly *what* this changes achieves in one short & meaningful phrase or sentences that summarize change purpose!  Include references (issue, related PR's). Ensure the commit description clearly explains your code modifications with the context.

  Our maintainers and team will review it with diligence; be open to revisions, discussions around approach and alternative implementations or even complete changes as the process progresses towards merge into production/main/trunk branches! After your code is successfully reviewed and accepted (approved through pull request review).



## 6. Coding Standards

Consistent code style makes projects maintainable and enjoyable to contribute to. Therefore our goal with our standards ensures we adhere with these best practices! Please follow these guidelines, so your changes align seamlessly and reduce code maintenance and conflicts in future work:

Code consistency includes indentation using either tabs or 4 spaces (not mixed together). Naming rules must align to project naming conventions - typically, using meaningful identifiers is key, such as variables/functions with snake_case and descriptive classnames! Keep the code clean! Remove all unused imports and code, and keep code comments up to date! 

Use static code analysis tool (like linter/style checker), if applicable or available in this repository to help catch code smells or violations and keep a standard! This may mean using Python's PyLint, JavaScript's ESLint (ESLint with AirBnB),  C/C++ with `clang-format`, depending on what's applicable! We want to keep it standardized as possible - follow what exists, don not reinvent wheel! This also makes reviews easier - easier reviews are good.

  Document your code. This means including appropriate inline code comments as required! Ensure the function's parameters and returned results, and also any exceptions, as needed and required, as necessary with docstrings for functions, with descriptions. This helps future developers to easily comprehend functionality in a quick and understandable method, as it makes code readable for anyone joining!

  All contributions must include proper test documentation (discussed below - see Testing Guidelines). Please adhere strictly! Many automated tools are capable of validating and checking standards! These are important and we value and prioritize them for the code to be accepted - they ensure that we're all in lockstep.



## 7. Testing Guidelines

Testing forms a significant piece for the health of a product as the project continues to develop - it helps guarantee a certain amount of functionality that continues with future changes! Ensure existing test passes and add new, focused tests! If not you might face issues when code goes down the development pipeline: 

We have an integrated test runner to ensure our test cases work properly - use our standard framework or tools and instructions provided with it, to set it up locally to be prepared and ready. If we use pytest (in python projects for example), ensure `pip install pytest`. Then `pytest` can get running in local console - and run test coverage, as available, using  `pytest --cov` and `cov coverage`. This can give insight for test completeness as well, but remember this tool is optional but highly advised and useful to get better test completion coverage.

We expect you write Unit Tests for new or heavily-changed features to make testing much easier. This is usually done through mock functions! Write tests at each component or class and function! We are happy for you if it also runs through some form of integration (if possible or required to make it run correctly!). It ensures that code components are all well coupled to run seamlessly in production and not fail at deployment/testing phases. Regression test incoming PR and test that no code regressions were caused by your PR/change in question

Remember, comprehensive test coverage improves the project robustness & stability!

## 9. Commit Message Guidelines

Commit messages form one of most crucial aspects - they describe change context to everyone. A standard set helps everyone easily identify what each individual commits are contributing towards the bigger picture in future development:

Use present tense to explain code change, so commit should describe change rather than describe an action - "Add functionality". Don limit to ~50 chars in length!

Adhere strictly with Conventional Commits:  Use the prefix to specify intent `feat:`  (`New Features`)  `fix:` (`Bug Fixes`),  `docs:` (`Documentation Changes`)  ,`chore:`(`Other`) and use description with concise, short message following! It provides structured and automated way to understand history - helps generate a Changelog as part releases

  Write commit description explaining exactly how the contribution changes codebase, what impact, benefits. As it should provide the information to other devs!

   Ensure messages adhere with a blank line, to ensure a clean, consistent, standard message format for easier reading - helps with automatic release tooling as they expect specific format/layout for commit description for automation!



## 10. Branching Model

Our team utilizes the standard trunk-based branch flow for branches as development progresses and releases come together! We want all code going back to one location.

All new changes, bug fixes are started from the main branches (`main` or develop). It keeps all branches synchronized in all environments.

  Always prefix your local/remote name of branch (i.e.:  If starting new fix on a bug, you can do something with `fix/xyz`  for example to quickly indicate the nature or purpose!)

Submit pull requests rather with descriptive messages (documented well in previous structure), to get reviewed. A dedicated set of project Maintainers (and core team!)  review changes and provide feedback (with detailed and specific). They ensure changes meet standards & align! They ensure functionality, test quality is adequate

 Be receptive & prepared and open for code/design feedback/suggestion as review continues, to work together to improve changes to align with overall standards, vision or direction (this includes docs too!).  If the feedback is approved or fixed as referred to (i.e - a meaningful code fix or documentation correction). A reviewer must recheck, verify change. When it satisfies all, they're able to merge into `main` or develop - after code merge and release process occurs, after all steps in pipeline (CI).
