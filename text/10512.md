## 3. Getting Started

To get started, it helps to set up the environment. First and foremost, it' the best to familiarize ourselves with our repo on the cloud, typically using GitHub (e g, forking, creating an accout if you do not have one). Clone down the relevant repository, so you have the files to work with. 

Once it is downloaded onto your machine, you' will likely want to install any needed prerequisites. Typically the `INSTALL.md file`, which should be included in the repo, provides details regarding dependencies such as Python, Javascript and required libraries. Please read through the file before continuing.  

Branching rules are crucial. When you have forked and cloned your own copy of the repository, create separate local branches for individual features or bug fixes. Name the branches following a convention such as: feature/new-feature, fix/bugfix and docs/documentation updates. Do not work directly on main.

Next, build the software using the project setup instructions.  This may mean executing the `npm install` (for frontend) or `setup.py install` ( for backends) commands, which installs all the required dependencies. The specific installation and build processes will vary depending on the codebase, so it' important to follow any provided instructions precisely.

## 5. Reporting Bugs

When submitting a new issue, please make your report as descriptive as possible, with a detailed explanation, and a link to GitHub if applicable, to provide context. It is best to provide a detailed, reproducible, clear, and step-by-step instructions.

A clear title is also a key component of your issue, and it should concisely describe the bug in an easily identifiable, clear language that can be quickly and directly read. Please ensure you are providing as much detail as necessary, which should allow our team to fully understand the scope of issues and resolve them quickly.

When reporting a bug, ensure you are checking that the issue has not yet been raised by a contributor before submitting a new issue. A thorough check of the issue tracker on the repository can help eliminate duplicates.

Finally, to aid the reproduction of the issue, consider adding a log file, if applicable for your specific case. A log provides valuable debugging information that can assist the developers in resolving the issue, as it can often highlight specific areas for further investigation as a starting point.

## 7. Pull Request Process

To submit a pull request, make sure you have branched using:
 ```bash
 git checkout - b feature/branch-name
  ```
Commit frequently with clear, descriptive messages. After committing, push the code from your feature branch to the fork.

Next, create a new pull request from your fork's branch to the Acme main or development branch. Write a detailed pull request description, outlining the intended functionality, the changes made, and any potential impact on existing features or the broader ecosystem.

Our maintain team reviews each PR thoroughly for style, functionality, and adherence to the established standards. They may provide feedback to improve the solution or identify potential issues. Please be receptive to their feedback, addressing issues and iterating on your code as necessary, so the integration will prove seamless and consistent with the existing ecosystem.

## 8. Coding Standards

We adhere to the PEP 8 style guide for Python, and the Google JavaScript Style Guide for JavaScript. These standards promote code readability and consistency, and contribute to a maintainable base of code. We encourage the use of code linters (flake8 for Python and ESLint for JavaScript) that ensure code follows the established guidelines.

To ensure consistency, we use pre-commit hooks to automatically check for formatting, linting, and type checking before a change is committed. This reduces the risk of introducing code that does not meet style requirements and helps maintain a high-quality base of code.   

We require detailed documentation for all functions, classes, and modules, ensuring that the code is easily understandable by both developers and end-users alike. Documentation should be accurate, comprehensive, and up-to-date to facilitate collaboration, and ensure code longevity and maintainability.      

We encourage contributors to use a consistent naming convention, and to avoid abbreviations and acronyms. Code should strive to be expressive and easy to understand, and avoid overly complicated or abstract logic that may hinder understanding across all contributors and users alike.  

## <font color = "red "> **Code must be PEP 8- compliant, using a pre- commit system that runs automatically before committing new content**

We expect tests to be well-written, readable, and comprehensive, to cover all the necessary functionality and edge- cases. All tests should be automated, meaning the tests must run without manual intervention, which facilitates continuous and reliable testing, and ensures quality across the entire project as a whole.      

We welcome the use of any testing frameworks, but it's best to adhere to the existing, widely recognized, frameworks, and tools, such as pytest, Jest, and similar frameworks to facilitate ease of contribution, and seamless maintenance.

## 11. Commit Message Guidelines

Clear and descriptive commit messages enhance traceability and collaboration across project stakeholders; a good one provides essential information on what happened, when, and how and adds a great degree of visibility across contributors in the code lifecycle and management phases.     

Commit messages should adhere to the conventional commits standard, following a concise title of at most 50 characters followed by a space, followed by a descriptive body if needed to describe a complex feature implementation in sufficient details to enable others in understanding what changes were deployed to resolve.

Prefix your commit messages with an appropriate type: `feat` for new features, `fix` for bug fixes, `docs` for documentation updates, `style` for formatting, `refactor` for code refactoring, `test` for test additions, or `chore` for build process changes or maintenance. The conventional commiits system allows easy integration in automation, tooling, and version tracking processes for seamless release management for a greater efficiency and productivity as we scale across the community

The use of concise messages improves collaboration. It facilitates efficient communication of intent for other developers who might potentially contribute. By ensuring everyone's code changes and updates follow these guidelines.    

Always write commit messages in the imperative mood - for example:  'Fix issue #42" rather than "Fixed issue #42"   to promote an efficient, simplified code flow and communication.     

## <font color="blue "> **All code submissions follow conventions for descriptive content for clarity. All PR and releases use conventional format to enhance efficiency. **

## 12. Branching Model

Our primary goal to keep `main`, the most current branch with the stable production state; as this is the most rigidly monitored code available at any stage in production for release cycles, this is an integral step towards our mission in its stability.

Each branch for any work will have names starting from prefixes that specify functionality; typically prefixed like  `feature`,  `fix`,`doc`, and so on; the prefix should always describe work in that branch. The prefix can be a short and clear name, and must never cause conflicts across the ecosystem or team of stakeholders working to maintain this ecosystem

Branching must only happen directly from either `main` for small, minor adjustments that can happen without impacting core workflows, and/or `develop` (a development version), as these are primary branches, to minimize disruption across core functionality, and allow a clear distinction to be created and maintained for any code in transit or for review before ultimately integrating the change and publishing or deployment into a new release.
