 ##5 Reporting Bugs ##

When a bug is found, it' s extremely valuable to report the issue on our issue tracker ([Link to GitHub issues page, etc ]). Provide a concise but comprehensive title summarizing the nature of the problem, enabling a quick grasp of what is going on with that specific bug.  

Include a detailed description, explaining the steps you took, what behavior you expected ( what you think would be a proper response) and what actually happened (what the software had returned to that specific action or process). This is a key to helping the issue to be understood, and thus resolved more promptly. 

To improve debugging, it is vital to provide a minimal, reproducible example! This involves providing the least amount of code to trigger the bug, which can often be copied and run independently to verify your description . This allows others to easily replicate, understand and verify the problem, speeding debugging and fixes!

Attach logs if available.  Relevant system configurations ( operating system, hardware, installed software) will also help diagnose and reproduce problems. Be mindful of privacy - do not include personal information, and sanitize the output if there are any sensitive details!

## 7 Pull Request Process ##

To submit your code change, create a branch named `feature/[feature- name ]`, `fix/[bug- description]` based on the type of changes being submitted, from ` origin/main ` or your forked repository! This branching helps to keep work isolated, and makes merging and tracking changes more easier.

Push branch to GitHub and create a new pull request (PR) to ` main ` . This creates a proposal for your code change, allowing maintain and contributors to see code and leave comments or suggestions!. Ensure your code adheres to the project coding and testing standards - described in sections 7 and 8.

Your submission will undergo code review, which means other members of the team will inspect, evaluate, and comment on proposed code changes!. Be ready to address any comments and feedback, incorporating suggestions and improving your contribution until approval received from one or more maintainers.

Once approved, pull request will merge into ` main ` . This incorporates your changes into the main codebase, making it available to all users and contributors, improving the platform' functionality, reliability, or usability! After merge, ensure branch is deleted, to keep the repo tidy!

##11 Commit Message Guidelines ##

Clean, informative commit messages make it a breeze tracking and auditing project histories and contribute significantly towards project clarity. Every commit should provide meaningful context for change introduced!  

We follow the [conventional commits spec or guidelines](link to guidelines)! The messages have format, ` type(scope): description `.  For ` type`, commonly accepted prefixes are: `feat` for features, `fix` for bug fixes, `docs` for documentation changes, `style` for code formatting changes, `refactor` for code restructures without changes. and so on!. The description briefly explains what changes introduced - it has be as brief and understandable as possible!

The subject line must start the message while body (separated a blank line from subject line) explains rationale in longer detail!. It is good practise include links to associated tickets. or other documents when necessary in messages - which ensures better communication, understanding of rationale changes!

Always include relevant context! Avoid ambiguous, vague commits such as "Updates"or fixes". Commit history needs to be readable, and informative - clear communication helps in long term understanding of evolution of codebase.
